## 1. Product Overview

NutriTrack is a minimalist iOS app for tracking daily intake goals — initially protein (grams) and water (fl oz or ml). Its defining UX is a beautiful set of animated "tracker wheels" on the home screen, each displaying three concentric visual layers that communicate progress against two thresholds (a minimum goal and a main goal). The design is built for one-handed daily use: quick log entry via a custom numeric keypad and preset buttons, a heatmap history view, and a clean settings screen for configuring goals and appearance.

**Core UX philosophy:**
- Let the tracker wheels carry all visual weight. Maximum whitespace everywhere else.
- Logging an intake should take under 3 taps.
- History should be understood at a glance via color-coded heatmaps.
- The app must support up to ~6 tracker types without any UI redesign.

```
│   (historical data)   │   │   goals, appearance, units  │
└───────────────────────┘   └────────────────────────────┘
```

### 3.2 Key Architectural Rules

1. **Tracker-generic everywhere.** No `switch tracker.id { case "protein": ... }` anywhere in the codebase. All views iterate `activeTrackers: [TrackerType]` using `ForEach`. Adding a new tracker type requires only: (a) adding a default to `TrackerType.defaults`, (b) a Core Data migration (if schema changes). Zero view changes.

2. **No hardcoded colors.** Every `Color(...)` in a view traces back to either a SwiftUI semantic color (`Color(.systemBackground)`) or a hex string read from `SettingsViewModel` → resolved by `ThemeColors`. The `Color(hex:)` extension handles parsing.

3. **AppStorage for preferences, Core Data for history.** Goals, units, display order, colors → `@AppStorage`. Log entries → Core Data. This is a strict boundary.

4. **ViewModels own all data access.** Views never call Core Data directly. Views read `@Published` properties and call ViewModel methods.

5. **Canvas-based wheel rendering.** `TrackerWheelView` uses SwiftUI `Canvas` for a single GPU draw pass rather than layered SwiftUI shapes. This avoids view hierarchy overhead for the most frequently animated element.

6. **Swift 6 concurrency.** All ViewModels are `@MainActor`. Background fetches (history) use `async/await` with `newBackgroundContext()`. `CoreDataStack` is `@unchecked Sendable`.

## 4. Data Layer

### 4.1 TrackerType (In-Memory Registry)

`TrackerType` is a pure Swift struct — not a Core Data entity. It lives in `@AppStorage` as JSON-encoded `[TrackerType]` under the key `"trackerConfigs"`.

```swift
struct TrackerType: Identifiable, Codable, Hashable {
    let id: String          // "protein", "water", "fiber", etc.
    let displayName: String
    let unit: String        // "g", "fl oz", "ml"
    let isBuiltIn: Bool
    var minimumGoal: Double
    var mainGoal: Double
    var displayOrder: Int
    var isEnabled: Bool
    var pieColor: String    // hex, e.g. "#E8601C"
    var ringColor: String
    var barColor: String
}
```

**Fraction calculation methods** (core to the wheel rendering):

| Method | Input range | Output |
|---|---|---|
| `pieFraction(for:)` | 0 → minimumGoal | 0.0 → 1.0 |
| `ringFraction(for:)` | minimumGoal → mainGoal | 0.0 → 1.0 |
| `overflowFraction(for:)` | mainGoal → 2×mainGoal | 0.0 → 1.0 |
| `percentOfMainGoal(for:)` | 0 → 2×mainGoal | 0.0 → 2.0 |

All fractions are clamped to [0, 1] — visual elements never overflow.

**Default values (first launch):**
- **Protein:** min 120g, main 160g — pie `#E8601C` (burnt orange), ring `#F4A261` (peach), bar `#E76F51` (coral)
- **Water:** min 80 fl oz, main 120 fl oz — pie `#48CAE4` (sky blue), ring `#0096C7` (ocean blue), bar `#023E8A` (navy)

### 4.2 QuickAddPreset

Also a pure Swift struct in `@AppStorage` as JSON under `"quickAddPresets"`.

```swift
struct QuickAddPreset: Identifiable, Codable, Hashable {
    let id: UUID
    var trackerID: String
    var amount: Double
    var label: String?   // e.g. "Protein shake", nil → shows "25g"
}
```

Built-in defaults: Shake (25g), Chicken (30g), Egg (6g) for protein; Glass (8 fl oz), Bottle (16 fl oz), Large (32 fl oz) for water.

### 4.3 LogEntry (Core Data)

The only entity in the Core Data schema. Stored in the App Group container (`group.com.fredericbahnson.nutritrack`) so a future widget can read it.

| Attribute | Type | Notes |
|---|---|---|
| `id` | UUID | auto-generated |
| `trackerID` | String | matches TrackerType.id |
| `amount` | Double | always in canonical unit (g or fl oz) |
| `timestamp` | Date | log time, used to bucket by calendar day |
| `note` | String? | reserved, not yet shown in UI |

**Safe accessors** (`LogEntry+Extensions.swift`) prevent nil crashes:
- `safeID` → UUID (or .init() if nil)
- `safeTrackerID` → String (or "" if nil)
- `safeTimestamp` → Date (or .distantPast if nil)

**Factory method:** `LogEntry.create(trackerID:amount:timestamp:note:in:)` encapsulates creation, sets all fields, and returns the new object without saving (caller saves via `CoreDataStack.shared.saveViewContext()`).

### 4.4 DailyTotal (Computed, Never Stored)

There is no `DailyTotal` entity or cache. Today's totals are computed by:
1. Fetching all `LogEntry` records with `timestamp` in `[startOfDay, startOfTomorrow)`.
2. Grouping by `trackerID`.
3. Summing `amount` per group.

This aggregation happens in `TodayViewModel.fetchTodayEntries()` and is always fresh — no stale cache risk. History aggregation in `HistoryViewModel` uses `NSFetchRequest` with a date range, also aggregated in-memory after fetch.

### 4.5 CoreDataStack

Singleton at `CoreDataStack.shared`. Key setup:
- **Persistent store URL:** App Group container → `NutriTrack.sqlite`
- **Merge policy:** `mergeByPropertyObjectTrumpMergePolicyType` — last writer wins per property (handles concurrent writes from background contexts)
- **Auto-merge:** `viewContext.automaticallyMergesChangesFromParent = true`
- **Background contexts:** `newBackgroundContext()` used by `HistoryViewModel` for async fetches without blocking main thread

## 5. View Models

All three ViewModels are `@MainActor` classes conforming to `ObservableObject`. They are instantiated in `NutriTrackApp` as `@StateObject` and injected via `.environmentObject(...)`.

### 5.1 TodayViewModel

**Responsibility:** Today's log entries and daily totals.

```swift
@Published var dailyTotals: [String: Double]      // trackerID → sum for today
@Published var todayEntries: [String: [LogEntry]] // trackerID → sorted entries
```

**Methods:**
- `fetchTodayEntries()` — NSFetchRequest with date predicate [midnight, tomorrow midnight). Groups and sums by trackerID. Always called from view `onAppear`.
- `addEntry(trackerID:amount:)` — creates LogEntry, saves, refetches.
- `deleteEntry(_:)` — deletes LogEntry, saves, refetches.
- `updateEntry(_:amount:)` — sets new amount, saves, refetches.

The refetch-after-mutation pattern ensures the UI always reflects the database truth.

### 5.2 SettingsViewModel

**Responsibility:** All user configuration. Acts as the single source of truth for tracker registry, goals, colors, presets, and display preferences.

Key `@AppStorage` bindings (JSON-encoded where complex):
```
"trackerConfigs"   → [TrackerType]  (goals, colors, display order, enabled state)
"quickAddPresets"  → [QuickAddPreset]
"waterUnit"        → "floz" | "ml"
"appearanceMode"   → "system" | "light" | "dark"
"dayResetHour"     → Int (always 0)
```

**Critical computed property:**
```swift
var activeTrackers: [TrackerType] {
    trackers.filter(\.isEnabled).sorted(by: { $0.displayOrder < $1.displayOrder })
}
```
This is the list every view iterates to build its tracker grid — the single source of display order truth.

**Tracker management methods:** `toggleTracker`, `updateGoals`, `updateColors`, `addCustomTracker`, `removeCustomTracker`, `reorderTrackers`, `updateTrackerName`, `tracker(for:)`.

**Water unit handling:** When the user switches units, `updateWaterUnitDisplay()` converts all water-related goals and presets from the old unit to the new unit so numerical values stay correct.

**Initialization:** `loadAll()` is called on init. If AppStorage is empty (first launch), defaults from `TrackerType.defaults` and `QuickAddPreset.defaults` are written.

### 5.3 HistoryViewModel

**Responsibility:** Historical data for the History screen.

```swift
@Published var selectedTrackerID: String
@Published var dailyTotals: [Date: Double]  // startOfDay → total
@Published var isLoading: Bool
```

**`fetchDailyTotals(from:to:) async`** runs on a background context:
1. Fetches all LogEntry records for selected tracker in date range.
2. Groups by `DateHelpers.startOfDay(for: entry.timestamp)`.
3. Sums amounts per day.
4. Publishes result back to `@MainActor`.

**`heatmapColor(for:tracker:themeColors:)`** delegates to `ThemeColors.heatmapColor(for:tracker:)`, which applies the three-tier color logic (green/blue/purple).

**`listData(for:)`** — filters `dailyTotals` to days with non-zero totals, sorts descending.

### 5.4 ThemeColors

`@MainActor ObservableObject` used as a pure read helper. Injected as `@EnvironmentObject`.

- `pieColor(for:)`, `ringColor(for:)`, `barColor(for:)` — reads hex string from `TrackerType`, returns `Color(hex:)`.
- `heatmapColor(for amount:tracker:)` → `HeatmapColors.color(for:tracker:)`:
  - 0: transparent/background
  - 0 < amount ≤ minGoal: green (`#4CAF50`), opacity 10%–100% proportional to fraction
  - minGoal < amount ≤ mainGoal: blue (`#2196F3`), opacity 10%–100%
  - amount > mainGoal: purple (`#9C27B0`), opacity 10%–100% (capped at 2×mainGoal)

## 6. Theme & Design System

### 6.1 Typography

All constants in `Typography.swift`. Two font families:

**SF Rounded** (numerics/display):
| Name | Size | Weight | Used for |
|---|---|---|---|
| `largeNumber` | 36pt | bold | Wheel center — primary amount |
| `mediumNumber` | 24pt | semibold | Wheel center — smaller display |
| `smallNumber` | 14pt | medium | Secondary numbers |
| `keypadDigit` | 28pt | medium | Keypad button labels |
| `keypadDisplay` | 48pt | bold | Entry field display number |

**SF Pro** (text/labels):
| Name | Size | Weight | Used for |
|---|---|---|---|
| `unit` | 13pt | regular | Unit labels in wheels |
| `label` | 15pt | medium | Tracker names, field labels |
| `caption` | 12pt | regular | Secondary info, percentages |

### 6.2 Color Rules

- **No hardcoded colors.** The only literal Color values allowed are SwiftUI semantic colors like `Color(.systemBackground)`, `Color(.label)`, `Color(.secondaryLabel)`.
- Tracker-specific colors (pie/ring/bar) always resolve through `ThemeColors` → `Color(hex:)`.
- Heatmap colors (green/blue/purple) are constants in `HeatmapColors` but are defined as hex strings, not literal SwiftUI colors, so they can be themed in the future.

### 6.3 Spacing Scale

4 / 8 / 16 / 24pt grid. Corner radii follow the same scale. No arbitrary magic numbers.

### 6.4 Dark Mode

`.preferredColorScheme()` is applied at the root `WindowGroup` level based on `@AppStorage("appearanceMode")`. All backgrounds and text use semantic SwiftUI colors that adapt automatically. Tracker and heatmap colors are user-chosen and render the same in both modes.

### 7.1 Layout Zones

```
┌────────────────────────────────┐
│                                │
│   TrackerGridView              │  ← ~66% of screen height
│   (animated tracker wheels)    │
│                                │
├────────────────────────────────┤

└────────────────────────────────┘
```

Tapping ⚙️ → Settings sheet. Tapping 📅 → History sheet. Tapping ＋ or any tracker wheel → Entry sheet.

### 7.2 TrackerGridView — Responsive Layouts

| Active trackers | Layout |
|---|---|
| 0 | `ContentUnavailableView` "No active trackers" |
| 1 | Single centered wheel, 80% of zone width |
| 2 | `VStack`, equal height (50% each) |
| 3 | Pyramid: 1 top (40% width, larger) + 2 bottom side-by-side (30% each) |
| 4 | `LazyVGrid` 2×2, equal size |
| 5+ | Scrollable 2-column `LazyVGrid` |

The user controls display order in Settings. `TrackerGridView` reads `settingsVM.activeTrackers` (already sorted by `displayOrder`) and renders in that order.

### 7.3 TrackerWheelView — Visual Design

Each tracker wheel is drawn in a single `Canvas` pass. From outside to inside:

```
    ┌─┐   ← overflow bar (8pt wide, right of ring)
  ╔═══╗
  ║ ◎ ║   ← ring (annular, ~20% of diameter, fills clockwise from 12)
  ║ ● ║   ← pie (circle, ~55% of diameter, fills clockwise from 12)
  ║95g║   ← center label (amount + unit, percentage below)
  ║59%║
  ╚═══╝
```

**Layer geometry:**
- `ringOuterRadius` = wheel.size / 2
- `ringInnerRadius` = ringOuter × 0.80 (20% thickness)
- `pieRadius` = ringInner − 2pt (2pt gap between pie and ring)
- Overflow bar: x = ringOuter + 8pt, width = 8pt, height = wheel diameter

**Fill semantics:**
- **Pie** fills as intake rises from 0 → minimumGoal. At minimumGoal, pie is fully filled.
- **Ring** begins filling when intake > minimumGoal and reaches full at mainGoal.
- **Overflow bar** begins filling when intake > mainGoal and reaches full at 2×mainGoal.

This gives the user a clear visual progression: "am I hitting my minimum?" (pie), "am I hitting my main goal?" (ring), "am I crushing it above goal?" (bar).

**Animations:**
- `withAnimation(.easeInOut(duration: 0.35))` wraps any `dailyTotal` change.
- `onAppear` triggers a 500ms easeInOut animation from 0 to actual value.
- Preset tap: quick pulse (scale 1.12 + opacity 0.6, 150ms easeOut, then reverse).

**Accessibility:**
- `.accessibilityLabel`: tracker display name
- `.accessibilityValue`: "{amount} {unit}, {percent}% of main goal"
- `.accessibilityHint`: "Double tap to log" (if tappable)

### 7.4 Entry Area (Bottom Sheet)

Presented as a sheet from main screen. Layout top to bottom:

1. **Tracker selector** (pill-tab row) — only shown if 2+ trackers active. Selects which tracker to log.
2. **Quick-add preset row** — horizontally scrollable pills filtered to selected tracker. Tap = instant log + dismiss.
3. **NumericKeypadView** — always visible inline (not a system keyboard, no keyboard avoidance needed).
4. **Today's entries list** — shows entries for selected tracker, with edit and delete per row.

**NumericKeypadView layout:**
```
        ┌─────────────────────────┐
        │       25.5              │  ← display number (48pt bold)
        └─────────────────────────┘
   ┌───┐ ┌───┐ ┌───┐
   │ 1 │ │ 2 │ │ 3 │
   ├───┤ ├───┤ ├───┤
   │ 4 │ │ 5 │ │ 6 │
   ├───┤ ├───┤ ├───┤
   │ 7 │ │ 8 │ │ 9 │
   ├───┤ ├───┤ ├───┤
   │ . │ │ 0 │ │ ⌫ │
   └───┘ └───┘ └───┘

```

Input constraints: max 1 decimal place, max 6 digits total, no leading zeros. "Log" button is disabled when field is empty or invalid.

**Today's log section** (within the entry sheet): lists all entries for selected tracker today. Each row: time (2:34 PM), amount + unit, edit button (pencil icon, opens `EditEntrySheet`), delete button (trash icon). Changes update the wheel in real time.

|---|---|---|

- `TabView` with `PageTabViewStyle` — horizontal swipe to navigate weeks.
- Range: 52 weeks back from today (1 year of history).
- Each page: 7 `HeatmapDayCell` squares arranged in a row.
- Column headers: M T W T F S S (abbreviated day names).
- Below each square: day number ("3", "4").
- Today's square: 1.5pt border outline.

- Swipe right = go back in time; swipe left = forward (toward today).

### 8.4 Monthly View

- `TabView` with `PageTabViewStyle`, vertical axis — swipe up for previous month, swipe down for next.
- Range: 24 months back.
- Calendar grid: 7 columns (Mon–Sun), variable rows.
- Header: "MMMM yyyy" (e.g., "March 2026").

### 8.5 List View

Toggle via the list icon in the bottom-left of the History screen. Replaces the heatmap with a plain `List` of all days with a non-zero total, newest first. Each row: date, colored dot (matching heatmap tier), total + unit. Tapping does nothing — display only.

## 10. App Logic & Data Flow

### 10.1 First Launch

1. `SettingsViewModel.loadAll()` is called in `init()`.
2. `@AppStorage("trackerConfigs")` is empty → writes `TrackerType.defaults` (protein + water, both enabled).
3. `@AppStorage("quickAddPresets")` is empty → writes `QuickAddPreset.defaults` (6 built-in presets).
4. `TodayViewModel.fetchTodayEntries()` runs → finds nothing → publishes empty totals.
5. Main screen renders: two tracker wheels at 0% fill.

### 10.2 Logging an Entry

1. User taps a tracker wheel or the ＋ button → `EntryAreaView` sheet appears.
2. If 2+ trackers: user selects which tracker via pill tabs.
3. User either taps a quick-add preset OR types a number on the keypad and taps "Log".
4. `TodayViewModel.addEntry(trackerID:amount:)` is called:
   - Creates `LogEntry` in viewContext.
   - Calls `CoreDataStack.shared.saveViewContext()`.
   - Calls `fetchTodayEntries()` → updates `dailyTotals`.
5. Views observe `@Published dailyTotals` → `TrackerWheelView` receives new value → `withAnimation(.easeInOut(duration: 0.35))` fills the wheel layers.

### 10.3 Editing/Deleting a Log Entry

1. Entry appears in today's log section of `EntryAreaView`.
2. Tap edit → `EditEntrySheet` with keypad pre-filled with existing amount.
3. Tap save → `TodayViewModel.updateEntry(_:amount:)`.
4. Or swipe/tap delete → `TodayViewModel.deleteEntry(_:)`.
5. Either path: save + refetch + animate wheel to new total.

### 10.4 Viewing History

1. User taps 📅 → `HistoryScreen` appears.
2. `onAppear`/`task` calls `historyVM.fetchDailyTotals(from:to:)` for the selected tracker, past 365 days.
3. Background context fetches all `LogEntry` records in range, aggregates by day, publishes `dailyTotals: [Date: Double]`.
4. `WeeklyHeatmapView` or `MonthlyHeatmapView` iterates dates and calls `themeColors.heatmapColor(for:tracker:)` per cell.
5. Switching tracker in the picker → `historyVM.selectedTrackerID` changes → new fetch runs.

### 10.5 Changing Settings

- **Toggle tracker on/off:** `settingsVM.toggleTracker(id:)` → flips `isEnabled` → saves JSON → `activeTrackers` computed property updates → views re-render via `@Published`.
- **Change goal:** `DecimalPadTextField` calls `settingsVM.updateGoals(for:min:main:)` on dismiss → updates tracker → saves JSON → wheel fractions recompute (since fraction methods read from TrackerType directly).
- **Change color:** `ColorPicker.onChange` → `settingsVM.updateColors(for:pie:ring:bar:)` → saves JSON → `ThemeColors` resolves new hex → wheels re-render.
- **Reorder trackers:** List drag → `settingsVM.reorderTrackers(from:to:)` → updates `displayOrder` fields → `activeTrackers` re-sorts → `TrackerGridView` re-renders.

## Appendix: Key Decisions (from DECISIONS.md)

- **TrackerWheelView uses Canvas**, not a ZStack of SwiftUI shapes. Single GPU draw pass, better performance for animated content.
- **Custom numeric keypad** is an inline view within the entry sheet, never a system keyboard. No keyboard avoidance needed.
- **Day boundary = midnight local time.** `dayResetHour` in AppStorage is reserved for future configurable cutoff.
- **Water unit canonical storage = fl oz.** All Core Data amounts stored in fl oz; display converts on the fly.
- **SF Symbols used:** `gearshape.fill` (settings), `calendar` (history), `list.bullet` (list view), `house.fill` (home), `clock.arrow.circlepath` (today's log), `delete.left.fill` (keypad backspace).
